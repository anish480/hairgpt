MoxieBuddy — Message Flow Map

How every message type travels from the Streamlit UI through the FastAPI backend to a Gemini-powered response.

System Architecture Overview

Chat interface, photo upload, option buttons, YouTube embeds

POST /chat, POST /photo/analyze, GET /health

Hybrid pgvector + FTS scoring from kb_chunks

Tool Definitions (passed to Gemini)

recommend_routine

Get the recommended Moxie product routine based on hair attributes and goals. Called when the user has shared enough about their hair type and what they need.

Parameters (OBJECT):
• hair_pattern — STRING, enum: straight | wavy | curly | coily (required)
• primary_goal — STRING, enum: frizz_control | wave_definition | curl_definition | damage_repair | general_care (required)
• is_chemically_treated — BOOLEAN
• is_colored — BOOLEAN

Execution: app/recommendations.py recommend_routine() — iterates RECOMMENDATION_RULES list, returns first match (routine name + enriched products from product_catalog.json). Falls back to "Rinse & Shine Duo".

get_product

Get details (name, price, link) for a specific Moxie product by its Shopify handle.

Parameters (OBJECT):
• product_handle — STRING (required), e.g. "gentle-cleansing-shampoo"

Execution: app/recommendations.py get_product() — looks up handle in PRODUCT_CATALOG dict loaded from data/product_catalog.json.

Flow 1 — Text Message (Regular Chat)

Text Message
  1. 1
    User types message in Streamlit

    scripts/demo_ui.py st.chat_input("Ask me anything about hair care...")

    The input is captured as prompt. If the user previously had a photo context stored in st.session_state.hair_context, the summary is prepended to the message as a [Hair photo analysis: ...] prefix.

    actual_message = "[Hair photo analysis: {summary}]\n\n{user_text}" # if photo context exists actual_message = "{user_text}" # otherwise
  2. 2
    HTTP POST to /chat

    scripts/demo_ui.py sends httpx.post(API_URL + "/chat", json={"message": actual_message, "history": api_history})

    POST /chat { "message": "My hair is so frizzy, help!", "history": [ {"role": "user", "content": "..."}, {"role": "assistant", "content": "..."} ] }
  3. 3
    FastAPI endpoint receives request

    app/main.py chat_endpoint(req: ChatRequest) → generates a session_id if empty, then delegates to the orchestrator.

    Calls: response_text, updated_history, options = await chat(req.message, req.history)

  4. 4
    RAG Retrieval — embed query + hybrid search

    app/orchestrator.py chunks = await retrieve(user_message, top_k=5)

    app/retrieval.py retrieve() does two things:

    4a. Embed the query: query_emb = (await embed([query]))[0] using app/llm.py embed() → calls Vertex AI text-embedding-005 with output_dimensionality=768.

    4b. Run hybrid SQL query against kb_chunks table:

    -- CTE 1: Vector search (cosine similarity) WITH vec AS ( SELECT id, content, chunk_type, source_id, product_refs, hair_types, 1 - (embedding <=> $1::vector) AS vec_score FROM kb_chunks WHERE is_active ORDER BY embedding <=> $1::vector LIMIT 30 ), -- CTE 2: Full-text search fts AS ( SELECT id, ts_rank_cd(content_tsv, plainto_tsquery('english', $2)) AS fts_score FROM kb_chunks WHERE is_active AND content_tsv @@ plainto_tsquery('english', $2) LIMIT 30 ) -- Combine with hybrid scoring SELECT ..., v.vec_score * 0.7 + COALESCE(f.fts_score, 0) * 0.3 AS score FROM vec v LEFT JOIN fts f ON v.id = f.id ORDER BY score DESC LIMIT 5

    Returns list of RetrievedChunk objects (id, content, chunk_type, source_id, product_refs, hair_types, score).

  5. 5
    Build system prompt with retrieval context

    app/orchestrator.py retrieval_context = format_retrieval_context(chunks)

    app/retrieval.py format_retrieval_context() formats each chunk as [Chunk Type] content, joined by --- separators.

    app/prompts.py build_system_prompt(retrieval_context) appends the retrieval context under a ## Relevant knowledge from Moxie's database header to the base SYSTEM_PROMPT.

    The system prompt contains: persona (MoxieBuddy), tone rules, brand guidelines, response length limits (60 words), product recommendation rules, escalation rules, few-shot examples, OPTIONS instruction, and the RAG context.

  6. 6
    Prepare contents and call Gemini 2.5 Flash

    app/orchestrator.py

    Converts history to list[types.Content] via _history_to_contents() (maps "assistant" → "model" role). Appends the new user message.

    config = GenerateContentConfig( system_instruction = system_prompt, tools = [TOOLS], # recommend_routine, get_product temperature = 0.7, max_output_tokens = 1024, thinking_config = ThinkingConfig(thinking_budget=0) ) resp = await client.aio.models.generate_content( model="gemini-2.5-flash", contents=contents, config=config )
  7. 7
    Tool-call loop (up to 3 rounds)

    app/orchestrator.py

    After each Gemini response, checks for function_call parts. If found:

    7a. Appends the model's response (with tool calls) to contents.

    7b. Executes each tool via _execute_tool(name, args):

    • recommend_routine → app/recommendations.py matches against RECOMMENDATION_RULES, enriches with product_catalog.json data
    • get_product → app/recommendations.py looks up product handle in PRODUCT_CATALOG

    7c. Appends tool results as Part.from_function_response and calls Gemini again.

    7d. Repeats until no more function calls or 3 rounds exhausted.

  8. 8
    Parse OPTIONS from response

    app/orchestrator.py display_text, suggested_options = _parse_options(response_text)

    Uses regex \n?OPTIONS:\s*(.+?)$ to extract pipe-separated options from the end of the response. Strips the OPTIONS line from display text.

    Input: "Winter dries hair out. Try a leave-in!\nOPTIONS: Tell me more|How do I use it?|Show me other products" Output: display_text = "Winter dries hair out. Try a leave-in!" options = ["Tell me more", "How do I use it?", "Show me other products"]
  9. 9
    Return response to UI

    app/main.py returns ChatResponse(response, session_id, history, suggested_options).

    scripts/demo_ui.py stores updated api_history and suggested_options in session state. Renders the response with render_message_with_videos() (extracts YouTube URLs for inline video embeds). Renders option buttons if any.

Flow 2 — Photo Upload

Photo Upload
  1. 1
    User uploads image via chat input

    scripts/demo_ui.py st.chat_input(accept_file=True, file_type=["jpg","jpeg","png","webp","heic"])

    Image bytes are read from the UploadedFile object. Displayed in the chat as a preview (st.image(image_bytes, width=300)).

  2. 2
    HTTP POST to /photo/analyze

    scripts/demo_ui.py sends multipart form: httpx.post(API_URL + "/photo/analyze", files={"file": (name, bytes, mime_type)})

  3. 3
    FastAPI validates and classifies

    app/main.py analyze_photo(file: UploadFile)

    Validates: content_type in {image/jpeg, image/png, image/webp, image/heic}, size ≤ 10 MB.

    Calls: classification = await classify_hair_photo(image_bytes, file.content_type)

  4. 4
    Gemini 2.5 Flash multimodal classification

    app/llm.py classify_hair_photo(image_bytes, mime_type)

    Sends the image + classification prompt to Gemini with structured JSON output schema:

    config = GenerateContentConfig( response_mime_type = "application/json", response_schema = HAIR_CLASSIFICATION_SCHEMA, max_output_tokens = 512, thinking_config = ThinkingConfig(thinking_budget=0) ) contents = Content(role="user", parts=[ Part.from_bytes(data=image_bytes, mime_type=mime_type), # the photo Part.from_text(text=CLASSIFY_PROMPT) # instructions ]) model = "gemini-2.5-flash"

    Classification schema fields:

    FieldTypePossible values
    hair_patternSTRINGstraight, wavy, curly, coily, unable_to_determine
    hair_pattern_codeSTRING1a–4c, unknown
    frizz_levelSTRINGnone, mild, moderate, severe
    damage_signsSTRINGnone, mild, moderate, severe
    lengthSTRINGshort, medium, long, very_long
    confidenceSTRINGhigh, medium, low
    notesSTRINGfree text
  5. 5
    Format classification into human-readable summary

    app/llm.py format_classification_summary(classification)

    Example output: "Based on your photo, you have wavy hair (around 2B), mild frizz, medium length. (Note: I'm not very confident in this assessment...)"
  6. 6
    Return classification + summary to UI

    app/main.py returns PhotoAnalysisResponse(classification=dict, summary=str).

    scripts/demo_ui.py stores classification and summary in st.session_state.hair_context.

    7. ↓↓↓ photo summary is now injected into the chat flow ↓↓↓
  7. 7
    UI constructs enriched message and sends to /chat

    scripts/demo_ui.py

    actual_message = "[User uploaded a hair photo. Analysis: {summary}]\n\n" + "Please respond to this hair analysis naturally." + "\n\nThe user also said: {user_text}" # if user typed text with photo

    This then follows the exact same flow as a text message (Flow 1, steps 2–9): POST /chat → orchestrator → retrieval → Gemini → tool calls → OPTIONS → response.

  8. 8
    Subsequent messages carry photo context

    On future text messages, if hair_context exists in session state and hasn't been injected into api_history yet, the UI prepends the summary:

    actual_message = "[Hair photo analysis: {summary}]\n\n{user_text}"

Flow 3 — Suggested Option Click

Option Click
  1. 1
    User clicks a suggested option button

    scripts/demo_ui.py — Options are rendered as Streamlit buttons from st.session_state.suggested_options.

    # Button rendering cols = st.columns(min(len(suggested_options), 3)) for i, option in enumerate(suggested_options): if col.button(option, ...): st.session_state.suggested_options = [] st.session_state._pending = option # set the option text as pending input st.rerun() # trigger rerun
  2. 2
    Pending option becomes the prompt

    On the next Streamlit rerun cycle, pending = st.session_state.pop("_pending", None) retrieves the option text and assigns it to prompt.

    The option text (e.g., "Recommend a routine") is used as the user message, exactly like typed text.

  3. 3
    Follows text message flow (Flow 1, steps 2–9)

    The option text goes through POST /chat → orchestrator → retrieval → Gemini → tool calls → OPTIONS → response, identical to a regular text message.

Flow 4 — Video Content in Responses

Video Content

How video tutorials get into the knowledge base

  1. 1
    Video metadata defined in catalogue

    data/videos.py — VIDEO_CATALOGUE is a hardcoded list of 9 VideoMeta dataclass objects. Each has: title, drive_id, description, products, routines, hair_types, topic_tags, and optional public_url (YouTube).

  2. 2
    Chunk generation

    data/videos.py generate_video_chunks() creates one KB chunk per video with content that includes the title, description, URL (best_url prefers YouTube over Drive), featured products, routine IDs, and applicable hair types.

    Example chunk content: "Tutorial video: Wavy Hair Routine Tutorial Complete 4-step Moxie wavy hair routine: shampoo, condition, leave-in... Watch here: https://drive.google.com/file/d/.../view Products featured: gentle-cleansing-shampoo, ultra-hydrating-conditioner, ... Routine: wavy_full Best for hair types: 2a, 2b, 2c"
  3. 3
    Ingestion into kb_chunks

    scripts/ingest.py loads video chunks from video_chunks.json (pre-generated JSON) alongside CX handbook chunks, hair type CSV chunks, and product CSV chunks. All chunks are embedded via embed_batch() (text-embedding-005, 768d) and inserted into the kb_chunks table with source_type = "video_tutorial".

How video URLs appear in chat responses

  1. 4
    Retrieval surfaces video chunks

    When a user asks "how do I use the curl cream?" or "show me the wavy routine", the hybrid search (Flow 1, step 4) can match video tutorial chunks via both vector similarity and FTS. The chunk content (including the video URL) is included in the system prompt's knowledge base section.

  2. 5
    Gemini includes video URL in response

    The system prompt instructs: "Share tutorial links from the knowledge base only when they genuinely add value." Gemini may include YouTube or Drive URLs in its response text.

  3. 6
    UI renders YouTube embeds inline

    scripts/demo_ui.py render_message_with_videos(text)

    Uses regex to detect YouTube URLs in the response text:

    # Regex patterns: _MD_YT_LINK = re.compile(r"\[([^\]]*)\]\((https?://...youtube...)\)") # markdown links _YT_RE = re.compile(r"(https?://...youtube.com/...([\w-]{11}))") # plain URLs # Processing: 1. Strip markdown link syntax: "[Watch here](https://youtube...)" -> "Watch here: https://youtube..." 2. Split text on YouTube URL matches 3. For each YouTube video ID found: st.video(f"https://www.youtube.com/watch?v={video_id}") 4. Surrounding text rendered as st.markdown()
